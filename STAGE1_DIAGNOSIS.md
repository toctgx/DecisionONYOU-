# DecisionONYOU 1단계 실행 결과 — 구조 진단 및 정리

## 1. 핵심 사용자 흐름 정리

### A. 온보딩 / 프로필 진입
1. 사용자가 앱에 진입한다.
2. 닉네임, 성별, 나이, MBTI, 비밀번호를 입력한다.
3. 동의 체크 후 앱 메인으로 들어간다.

현재 판단:
- 프로토타입에서는 유효함
- 실제 서비스에서는 비밀번호 입력 UI는 있지만 진짜 인증은 아님

결론:
- 유지하되, 추후 실제 인증 구조로 대체 필요

---

### B. 홈 피드에서 질문 소비
1. 사용자가 홈 피드에 진입한다.
2. 질문 유형에 따라 BUY/PASS 또는 THIS/THAT 카드를 본다.
3. 제한 시간 내 투표하거나 넘긴다.
4. 결과 미니바와 댓글을 볼 수 있다.

현재 판단:
- 앱의 가장 중요한 핵심 흐름
- 가장 잘 살아있는 기능

결론:
- 반드시 유지
- 2단계, 3단계 모두 이 흐름을 기준으로 보존해야 함

---

### C. 질문 등록
1. 사용자가 ASK 탭으로 이동한다.
2. 질문 타입(single / compare)을 고른다.
3. 이미지를 1장 또는 2장 업로드한다.
4. 캡션 입력 후 질문 등록한다.

현재 판단:
- 핵심 생산 흐름
- 이미지 업로드 + 질문 등록은 서비스 핵심 기능

결론:
- 반드시 유지
- 향후 스토리지/DB 구조 설계의 중심 기능

---

### D. 내 페이지 / 내 활동 확인
1. 사용자가 MY 탭으로 이동한다.
2. 내 질문 수, 내 투표 수를 본다.
3. 내가 올린 질문 목록, 참여 내역을 확인한다.
4. 일부 프로필 수정 시도를 한다.

현재 판단:
- 유저 유지에 도움이 되는 보조 흐름
- 다만 일부 기능은 불완전함

결론:
- 축소 유지
- MVP에서는 간단한 내 질문/내 투표만 남겨도 충분

---

### E. 댓글 / 신고 / 관리자 흐름
1. 홈에서 댓글을 작성하거나 댓글 목록을 본다.
2. 신고 버튼으로 게시물을 신고한다.
3. ADMIN 탭에서 신고 처리, 로그, 통계, 금지어, 공지 등을 조작한다.

현재 판단:
- 운영 관점을 미리 넣은 것은 장점
- 하지만 지금 구조에서는 전부 클라이언트라 실제 운영 기능으로서의 신뢰성 없음

결론:
- 댓글, 신고는 유지
- 관리자 기능은 대폭 축소 후 추후 서버 권한 구조로 재설계

## 3. 유지할 기능 / 제거 후보 기능 분류

### 반드시 유지할 기능
1. 질문 등록
2. 이미지 업로드
3. 홈 피드 소비
4. BUY/PASS 투표
5. THIS/THAT 투표
6. 댓글
7. 신고

이유:
- 이 앱의 정체성을 구성하는 핵심 루프이기 때문

### 축소 유지할 기능
1. 프로필 입력
2. 내 질문 목록
3. 내 투표현황
4. 공지 표시
5. 금칙어 검사

이유:
- 있으면 좋지만, 현재 구현은 실서비스형이 아님
- MVP에서는 간단화가 필요함

### 제거 또는 후순위 전환 후보
1. 관리자 통계 상세 UI
2. 관리자 로그 상세 UI
3. 로컬 기반 차단/경고 시스템
4. 복잡한 큐/반복 모드 로직 일부
5. 세부 공지/금지어 운영 UI

이유:
- 현재 구조에서는 유지비용 대비 가치가 낮음
- 실서비스에서는 서버 권한/DB 전환 없이는 의미가 약함

## 4. 기술 부채와 위험 요소 정리

### 4-1. 단일 파일 구조
현황:
- HTML/CSS/JS가 모두 `index.html` 하나에 몰려 있음
- 함수 수가 많고 상태 변수도 전역에 넓게 퍼져 있음

영향:
- 수정 범위 추적 어려움
- 기능 간 결합도 높음
- 테스트와 유지보수 어려움

우선순위:
- 최상

---

### 4-2. localStorage 중심 구조
현황:
- 프로필, 질문, 투표, 댓글, 신고, 관리자 설정 등이 전부 브라우저 로컬 저장소에 있음

영향:
- 멀티유저 불가
- 데이터 공유 불가
- 캐시 삭제 시 데이터 손실
- 운영 기능 신뢰성 없음

우선순위:
- 최상

---

### 4-3. 가짜 인증 구조
현황:
- 비밀번호 입력 UI는 있으나 실제 인증/세션 검증은 없음

영향:
- 사용자 식별 신뢰 불가
- 보안 기능이 있는 것처럼 보이지만 실제론 없음

우선순위:
- 상

---

### 4-4. 관리자 권한의 프론트 노출
현황:
- 관리자 기능이 클라이언트 코드 내부에 그대로 존재함

영향:
- 실서비스에서 악용 가능
- 운영 기능 보호 불가

우선순위:
- 상

---

### 4-5. 상태와 UI 렌더링의 강한 결합
현황:
- 상태 조작과 DOM 렌더링이 같은 파일에서 긴밀하게 뒤섞여 있음
- 이벤트 핸들러와 상태 변경 로직이 분리되지 않음

영향:
- 구조 분리 시 어려움 증가
- 버그 발생 시 원인 추적 어려움

우선순위:
- 상

---

## 5. 리팩터링 우선순위 확정

### 우선순위 1
- HTML / CSS / JavaScript 분리
- 단일 파일 구조 해소

### 우선순위 2
- 상태/스토리지/렌더링 책임 분리
- 최소한 `storage`, `ui`, `admin`, `app` 수준으로 관심사 분리 준비

### 우선순위 3
- MVP 핵심 흐름 기준으로 기능 재구성
- 질문 등록 / 피드 / 투표 / 댓글 / 신고 중심

### 우선순위 4
- 인증/권한/멀티유저가 가능한 백엔드 방향 결정

### 우선순위 5
- 운영 기능을 서버 권한 구조로 이관

## 6. MVP 범위 재확정

### 1차 MVP 권장 범위
- 사용자 식별(간단 로그인 또는 익명 + 닉네임)
- 질문 등록
- 이미지 업로드
- 피드 확인
- 투표
- 댓글
- 신고

### 1차 MVP에서 과감히 단순화할 부분
- 관리자 통계
- 관리자 로그 상세 UI
- 로컬 차단/경고 세부 기능
- 복잡한 반복 모드/큐 구조

---

## 7. 비서 판단 요약
이 앱의 핵심은 분명하다.

**사람이 고민을 올리고, 다른 사람이 빠르게 판단해주는 경험**이다.

그래서 1단계에서 실제로 정리된 결론은 다음과 같다.

1. 지켜야 할 코어는 명확하다.
2. 현재 구조는 서비스형이 아니라 로컬 프로토타입형이다.
3. 따라서 다음 단계에서는 기능 추가가 아니라 구조 분리가 우선이다.
4. 관리자/인증/운영 기능은 지금 그대로 키우면 오히려 독이 된다.
